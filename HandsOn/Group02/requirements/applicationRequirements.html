  1. Introduction

 

1.1 Purpose

 

Traffic Accident System (TAS) is a system to give the best way to use  the Madrid  transportation  for a final user indicating which is  the safest  route to a final destination considering historical  data  of accident happened  in Madrid.

The initial release also will show  to  emergency system critical points where accidents had  happened according  conditions to help them to be prepared for a faster response and also work on prevention to avoid  accidents.

 

1.2 Document Conversions

 

Sections are headed in blue  and subsections are headed in bold.

 

1.3 Project scope

 

This document describes the application requirements for traffic     accident system. The intended audience of this document exclusively includes the developers, the testers, and the end-users of the system.

 

1.4 Overview

 

The remainder of this document will be used to specify the requirements of the system.

 

 

 

 

  1. Overall Description

2.1 Product Perspective

 

The Dirección General de la Policía Municipal has access to a staggering amount of data that has been recorded since 2010 to 2018   relative to Traffic accidents in the City of Madrid with victims and / or damages to heritage. Data is representative of one record per person involved in the accident (drivers, travelers, pedestrians, witnesses, etc.)

 

TAS borns as a project for  asignature  Open Data and Knowledge Graphs  of the  Universidad Politecnica de Madrid coursed through the Masters in Data Science  in order to provide Madrid transportation users  and Madrid emergency systems a  way to analyze this data in an efficient, effective, expandable, and user friendly way.

 

2.2      Product Features

 

  TAS will provide the following features

 

  1. Flexible: By building TAS as a hybrid application, the user is able to connect to whatever capabilities the mobile platform provides through a browser that's embedded in the app.

 

  1. Reports: Once a zone or destination is given, TAS will display a list of the fastest AND safest routes based on weather and road conditions, with a comprehensive report of these conditions. TAS will also provide a historical reports of accidents that had happened on the routes.

 

  1. GoogleMaps: By using Google Maps, TAS will provide a user friendly and effective way of selecting the best route towards a destination, most specially for emergency systems. TAS will provide the safest routes with the least probability of accidents considering weather, road conditions and time of day.

 

  1. User friendly and intuitive: TAS will allow users to operate the system with the most minimum number of moves. TAS will record user behavior and will auto-predict user preferences. The more it is used, the more auto-prediction is fine-tuned.

 

  1. Voice Command: TAS can operate on voice command which will be very useful for emergency and security vehicles.

 

 

2.3      User Classes and Characteristics

 

The current user class consists mostly of car drivers,  bike drivers, pedestrian, Scooter drivers  that have a need to find the safest way to arrive to a destination, but may also  include  transit officers, emergency  systems to finding the  critical zones  where an accident may happen according to conditions.

 

 

Because TAS does not currently alter any data there will likely be little to no restrictions on who is authorized to use the system.

 

2.4    Operating Environment

 

TAS is a  mobile and web based application and will function properly in any mobile operating system and web browsers.

  1. System Features

 

3.1      User Friendly

 

  3.1.1 Description

 

Google Maps will be  used in order to make the application user friendly while maintaining the organization in report selection.

 

3.1.2 Stimulus/Response Sequences

 

The user input his  source and destination   in a combobox or select a zone in the city and then click on  an icon, they will get a report  about the safest way to take showing the probabilities on Google Maps.

 

3.1.3 Functional Requirements

 

REQ-1:          The user must be using one of the following browsers: Google Chrome, Mozilla Firefox, Safari or Opera.

 

 

 

 

3.2    Reports

 

  3.2.1 Description

 

After the user inputs his journey  they can select the type of report that they would like to view. Because the system’s purpose is to help select the best option with less probabilities of an accident.

 

Also TAS is   focus to help emergency systems  to be  more ready on critical points where most of the accidents happens, and also to do more prevention  before the accidents happens.

 

3.2.2 Stimulus/Response Sequences

 

When the report is generated the user can toggle checkboxes that will add and remove data sets, toggle between line and dot charts, and show probabilities.

 

3.2.3 Functional Requirements

 

REQ-1:          The user must be using one of the following browsers: Google Chrome, Mozilla Firefox, Safari or Opera.

 

 

3.3    Hybrid application

 

  3.4.1 Description and Priority

 

The project is designed as a hybrid application so final users  does not have to be concerned about platforms in  order to use the system.

 

3.4.3 Functional Requirements

 

REQ-1:          The user must be using one of the following browsers: Google Chrome, Mozilla Firefox, Safari, or Opera.

  1. Other nonfunctional requirements

 

5.1    Performance Requirements

 

TAS will need to be able to retrieve large amounts of data and process it  quickly. The current the acceptable limit from when data is requested to when the report is displayed is five seconds.

 

5.2      Safety Requirements

 

  TAS is not currently a safety risk as it doesn’t have rights to alter the database.

 

5.3      Security Requirements

 

TAS does not have any security risks except potential defacing of the website which must be prevented. For required text fields TAS does not still has privileges to perform an alter  on the database.

 

5.4      Software Quality Attributes

 

  Availability: TAS needs to be constantly running after deployment.

  Usability: TAS must have an intuitive user interface.

  Expandability: TAS must be easly expanded to incorporate more charts and

                            features
